## 数据流转

---

1. 底层逻辑：**打包请求 --> 互联网送达 --> 服务器拆包处理 --> 打包响应**。
   - 错误码：
     - 4xx = **你（客户端）打包错了** --> 服务器看不懂或没权限（你该请求就能好）
     - 5xx = 服务器自己处理时炸了 --> 你请求没问题，但服务器内部出故障（只能等或换接口）
2. 整个过程就像寄快递：写地址、写信（请求）、贴邮票（headers）、放内容（body）；快递公司（HTTP）送到；对方签收、处理、回寄（响应）。

## HTTP 协议

#HTTP

---

1. HTTP 设计的根本目的是什么？
   答案：它是互联网世界的“**标准点菜普通话**”，专门用来解决“我想买什么”和“店家给我什么”之间的沟通问题。
2. HTTP 协议就是“**点菜规则**”：保证”点菜单“和对应的“菜”在全世界任何地方都可以被正确的理解和交付的规则。
   3. **GET = “问路”**：你只问，不带东西，地址写在信封（URL）上，大家都可以看到。问 10 次都一样（幂等）。
   4. **POST = “寄包裹”**：你带行李（body）给对方，箱子里可以放很多东西，也可以让对方“改库存”。寄 10 次结果可能都不一样。

6. 核心区别：
   7. GET ：**GET = 伸手要东西（只看不给）**，**只索取，不提交，内容和状态完全不变**。 无 Body、参数在 URL 后面、适合“查”，可缓存、安全（是**语义约定**，不是技术强制。用 GET 删除数据 technically 可行但违反规范）。
   8. POST ： **POST = 递东西过去（又看又给）**，**要提交信息，服务器的状态会改变**。有 Body、适合“增/改”，POST 响应**可以**被缓存，默认不缓存，需要设置 Cache-Control 头。

## RESTful API

---

### RESTful API 是什么？

**RESTful API = 一套“标准化服务窗口”的规矩，让全世界的系统能像乐高积木一样互相拼接。**

**规矩**：

1. **所有东西都是”资源”，有统一编号**

| 生活例子                       | RESTful             |
| ------------------------------ | ------------------- |
| 快递柜每个格子有编号           | 每个数据有统一 URL  |
| `快递柜 - A区 - 03 号格`       | `/api/packages/A03` |
| `图书馆 - 3楼 - B区 - 025号书` | `/api/books/025`    |

**核心：不管是什么东西，都能用一个“地址“找到它。** 2. **操作方式只有 4 种，用 HTTP 方法区分**

| 你想做的事             | HTTP 方法 | 生活比喻           |
| ---------------------- | --------- | ------------------ |
| **查看**快递柜里有什么 | GET       | 看格子编号，不打开 |
| **存放**新包裹         | POST      | 把新包裹放进空格子 |
| **修改**包裹信息       | PUT       | 把包裹里的东西换掉 |
| **删除**包裹           | DELETE    | 把格子里的东西扔掉 |

**核心：不管怎么操作，只有这四种动作，不用发明新花样。** 3. **无状态，每次都说清楚**

| 有状态（累）                   | 无状态（轻松）                       |
| ------------------------------ | ------------------------------------ |
| 服务员要记住”你是第三桌的客人“ | 你每次都说”我是 3 桌的，要加冰“      |
| 你第二次去要说”还是老样子“     | 你每次都说”我要牛肉面，微辣，不要葱“ |
| 服务员记错就乱套               | 每次独立，不会记混                   |

**核心：每次请求都自带完整信息，不依赖之前的记忆。**

### RESTful 的 5 个核心特点

| 特点         | 大白话解释               | 生活比喻                           |
| ------------ | ------------------------ | ---------------------------------- |
| **资源导向** | 所有东西都有统一"地址"   | 就像每个快递柜格子有编号           |
| **统一接口** | 只用 GET/POST/PUT/DELETE | 就像只有"取/存/改/删"4 个按钮      |
| **无状态**   | 每次请求自带完整信息     | 就像每次点菜都说清楚，不靠服务员记 |
| **可缓存**   | 相同请求可以存起来复用   | 就像常点的菜可以提前备好           |
| **分层系统** | 中间可以加各种"中转站"   | 就像快递可以经过多个中转站         |

### RESTful 要解决什么问题？

答案：让 API 像网址一样直观，看到 URL 就知道在操作什么资源。

```txt
传统API：/api?action=getUser&id=1&token=xxx ❌ 看不出在干嘛
RESTful：GET /users/1 ✅ 一眼看懂
```

1. 概念：RESTful 不是新语言，而是一种“聪明点菜方式”--把一切东西的当成“资源”（像菜单上的菜），用 URL 代表资源，用 HTTP 方法决定你要干嘛。
2. 经典规则：
   - GET /posts --> 拿所有帖子
   - GET /posts/1 --> 拿第一个帖子
   - POST /posts --> 新建帖子
   - PUT /posts/1 --> 全量更新第一个
   - DELETE /posts/1 --> 删除
   - 优点：URL 一看就知道是什么，方法一看就知道要干什么，全球都统一
3. **补充知识**： 现在很多公司还在用 REST，但 GraphQL 和 REST 现共存（一次请求就能拿多给资源，，像“点一桌菜”）。
4. **Demo**：

   - 用同一个免费测试 API（它就是标准的 RESTful）：

   - GET https://jsonplaceholder.typicode.com/users → 拿所有用户（资源列表）
   - GET https://jsonplaceholder.typicode.com/users/1 → 拿用户 1（单个资源）
   - POST https://jsonplaceholder.typicode.com/users + Body（同上面 JSON）→ 创建新用户。
     看 URL 和 Method 就知道在操作什么资源，这就是 RESTful 的灵魂。

## JSON 数据格式的结构

---

1. JSON 要解决什么问题？
   答案：让人类和机器都能轻松读写的数据格式。
2. 概念：JSON 就是“**用大括号写的超级表格**”，电脑最爱吃。
3. **规则**：
   - {} = 对象（一个对象）
   - [] = 数组（一堆东西）
   - "key":value 键值对（key 必须用双引号）
4. 典型结构：

```JSON
	1.{
  "name": "仕杰",
  "age": 25,
  "isStudent": false,
  "hobbies": ["编程", "旅行"],
  "address": {
    "city": "Los Angeles",
    "zip": "90001"
  }
}
```

5. **服务器解析 JSON 的过程**：收到请求 --> 读取 Body --> 按 JSON 规则逐字符解析 --> 发现非法逗号 --> 返回 400
6. **补充知识**：以前用 XML （一大堆标签），现在 JSON 胜出是因为快、轻、人类可读。注意：中文必须 UTF-8，否则乱码。
7. **Demo**：用上面任意一个 GET 请求，返回的就是 JSON。 在 Postman 右边面板你会看到一堆 {} []，鼠标点开就能看到每一层的 key。 试着自己写个错的 JSON（多一个逗号）再 POST，看报 400，就懂了。

## 接口测试实战

---

1. 接口文档：
   2. 请求
   3. 响应：
      - 200：成功
      - 400 Bad Request：请求语法/格式有问题
      - 401：没有身份凭据
      - 403：权限问题
      - 404：接口地址写错/资源不存在
      - 405：用了 GET 但接口只允许 POST
      - 415：Content-Type 写错（该传 JSON 却传 form）
      - 422：格式对但业务逻辑不对（年龄 = -5)
      - 429：调用太频繁（超限流）
      - 500：服务器内部未知错误
      - 502：后端服务挂了（Nginx 前面代理出问题）
      - 503：访问时间过长
      - 504：服务器处理太慢，超时了。
