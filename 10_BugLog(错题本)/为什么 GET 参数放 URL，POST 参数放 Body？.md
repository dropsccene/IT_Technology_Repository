1. **设计的初衷**（**Semantics**）
	1. **GET**（**获取资源**）：语义是”**查询**“。就像图书馆里找书，只需要通过索引卡就行。URL 代表着资源的”地址“，参数则是对这个地址的限定（如：`?page = 2`）。既然是资源定位的一部分，参数理应出现在 URL 中。
	2. **POST**（**提交**/**处理资源**）：语义是”发送数据“。就像寄信一样，URL 是地址，而具体内容则相当于本体，应该塞进”信封“内（即 Request Body）。
2. **缓存与幂等性**（**Caching & Idempotency**）
	1. 这是两者最大的技术区别之一：
		1. **GET 的可缓存性**：由于 GET 参数在 URL 中，整个 URL （含参数）可以作为缓存的唯一标识（Key）。浏览器、代理服务器（CDN）和反向代理（Nginx） 都可以缓存特定的搜索结果。如果参数在 Body 里，缓存系统讲难以识别请求的差异。
		2. **书签与历史记录**：URL 中的参数运行用户将搜索结果”收藏“为书签或分享给他人。如果参数在 Body 中，用户分享的只是一个不含参数的空链接。
3. **安全性与隐私 (Security & Privacy)**
	1. 虽然 POST 并不比 GET 绝对安全（都需要 HTTPS），但在隐私保护上 POST 明显占优：
		- **日志泄露：** URL 会被记录在浏览器的历史记录、Web 服务器访问日志以及代理服务器日志中。如果 GET 用于发送密码，密码会以明文形式出现在所有日志里。
		- **Referer 泄露：** 当你从一个页面点击跳转到另一个页面时，当前页面的 URL（含 GET 参数）会通过 `Referer` 请求头发送给第三方网站。POST 数据则不会被这样传递。
4. 数据长度与类型的限制
	- **GET 的长度限制：** 尽管 HTTP 协议本身不限制 URL 长度，但浏览器（如早期的 IE 限制为 2083 字符）和 Web 服务器都会设置 URL 长度上限，以防止恶意攻击或内存溢出。
	- **POST 的灵活性：** Request Body 可以承载巨大的数据量（几 GB 的视频文件），并且支持多种编码格式（JSON, XML, 二进制流, 多部分表单数据）。URL 只能接受 ASCII 字符编码，处理复杂对象非常困难。
5. 总结对比表

| 特性        | GET (URL)     | POST (Body)        |
| --------- | ------------- | ------------------ |
| **主要目的**  | 获取、搜索数据       | 提交、更新、创建数据         |
| **可见性**   | 参数在 URL 中完全可见 | 参数在 Body 中，普通用户不可见 |
| **缓存**    | 可被浏览器和 CDN 缓存 | 默认不缓存              |
| **后退/刷新** | 无害            | 浏览器通常提示“确认重新提交表单”  |
| **数据类型**  | 仅限 ASCII 字符   | 无限制（支持二进制、JSON等）   |
| **安全性**   | 较低（易在日志中泄露）   | 较高（不会留在历史记录中）      |
 补充：GET 可以带 Body 吗？
从技术上讲，**可以**。HTTP 规范并没有禁止 GET 请求携带 Body。但是：

1. **很多服务器会直接忽略 GET 的 Body**。
    
2. **缓存中间件无法处理带 Body 的 GET**。
    
3. **不符合 RESTful 规范**。 因此，虽然 Elasticsearch 等某些特定工具会使用带 Body 的 GET（为了发送复杂的查询 JSON），但在通用 Web 开发中并不推荐这样做。
