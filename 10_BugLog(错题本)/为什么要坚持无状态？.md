**无状态 = 每次请求都”自带身份证和说明书“，服务器不用记你是谁、不用记你之前做过什么。**

**为什么要这样？因为“记性越好，死得越快”！**

### 什么是“有状态”vs“无状态”

**有状态 = 服务器要“记性”**

| 场景         | 有状态的做法                           |
| ------------ | -------------------------------------- |
| 你第一次登录 | 服务器记住："哦，这是张三，登录了"     |
| 你第二次请求 | 服务器查记忆："哦，这是刚才登录的张三" |
| 你第三次请求 | 服务器再查记忆..."                     |

**问题：服务器要维护一个“记忆本”，记录每个用户的状态**。

---

**无状态 = 每次都说清楚**

| 场景         | 无状态的做法                         |
| ------------ | ------------------------------------ |
| 你第一次登录 | 拿到一个"通行证"(Token)              |
| 你第二次请求 | 带上通行证："我是张三，这是通行证"   |
| 你第三次请求 | 再带上通行证："我是张三，这是通行证" |

**服务器不用记，每次只看通行证，有效就处理，无效就拒绝。**

---

### 为什么要无状态？5 个核心理由

**理由 1：服务器可以随便“复制粘贴”**

**有状态的坑**

```text
用户A → 服务器1(记得A)
用户B → 服务器2(记得B)
用户C → 服务器3(记得C)
如果服务器1挂了：
❌ 用户A的"记忆"没了，A得重新登录
❌ 用户A的请求不能转到服务器2，因为2不记得A
```

**无状态的好处**

```text
用户A → 服务器1(不记A，只看Token)
用户B → 服务器2(不记B，只看Token)
用户C → 服务器3(不记C，只看Token)
如果服务器1挂了：
✅ 用户A的请求自动转到服务器2，2看Token一样处理
✅ 用户可以随便被分配到任何服务器
✅ 随时可以增加/减少服务器，不用同步"记忆"
```

**比喻**：

| 有状态                                       | 无状态                                       |
| -------------------------------------------- | -------------------------------------------- |
| 去银行必须找同一个柜员，因为只有他记得你的事 | 去银行可以找任何一个柜员，每次出示身份证就行 |
| 那个柜员请假，你就办不了业务                 | 任何柜员都能办，谁在都行                     |

---

**理由 2：服务器不用“背黑锅”**

**有状态的坑**

| 问题                            | 后果                 |
| ------------------------------- | -------------------- |
| 服务器要存 100 万用户的登录状态 | 内存爆了             |
| 服务器重启                      | 所有用户得重新登录   |
| 服务器集群要同步状态            | 网络延迟、数据不一致 |
| 某个用户状态被篡改              | 安全漏洞             |

**无状态的好处**

| 好处           | 说明                         |
| -------------- | ---------------------------- |
| 服务器重启     | 用户不受影响，Token 还有效   |
| 不用存用户状态 | 内存省下来处理请求           |
| 不用同步状态   | 集群随便扩，不用互相"对暗号" |
| 状态在客户端   | 服务器只验证，不存储         |

**比喻**：

| 有状态                                     | 无状态                             |
| ------------------------------------------ | ---------------------------------- |
| 饭店服务员要记住每桌客人点了什么、吃到哪了 | 饭店服务员只看小票，小票在客人手里 |
| 服务员换班，得交接"3 桌吃到一半了"         | 换班无所谓，看小票就行             |
| 服务员记错了，菜上错了                     | 小票写清楚了，不会错               |

---

**理由 3：请求可以“随便重试”**

**有状态的坑**

```text
你点了一次"付款" → 服务器记住了"处理中"
你网络不好，又点了一次 → 服务器懵了：
"这人是要付两次款吗？还是刚才没成功？"
```

**无状态的好处**

```text
你点了一次"付款" → 带上Token和订单信息
你网络不好，又点了一次 → 服务器看订单号：
"哦，这个订单已经付过了" → 直接返回结果，不重复扣款
```

**比喻**：

| 有状态                                               | 无状态                                     |
| ---------------------------------------------------- | ------------------------------------------ |
| 你给客服打电话，必须找同一个人，因为只有他记得你的事 | 你给客服打电话，找谁都行，每次报订单号就行 |
| 电话断了，再打过去得重新说一遍                       | 电话断了，再打过去报订单号继续             |

---

**理由 4：缓存系统能“帮大忙”**

**有状态的坑**

| 问题                                         | 后果                         |
| -------------------------------------------- | ---------------------------- |
| 每个用户的请求都不一样（因为服务器要查状态） | 缓存没法用，每次都得查服务器 |
| 缓存了 A 的请求，B 来用不了                  | 缓存命中率低                 |

**无状态的好处**

| 好处                     | 说明           |
| ------------------------ | -------------- |
| 相同请求返回相同结果     | 可以大胆缓存   |
| 缓存服务器不用管用户状态 | 缓存层简单高效 |
| 减轻源服务器压力         | 性能提升       |

**比喻**：

| 有状态                             | 无状态                                     |
| ---------------------------------- | ------------------------------------------ |
| 图书馆每本书的借阅状态要实时查系统 | 图书馆每本书上贴个标签，一看就知道能不能借 |
| 每次都要问管理员                   | 自己看标签就行                             |

---

**理由 5：调试和监控“一目了然”**

**有状态的坑**

```text
用户说"我的请求失败了"
服务器查日志："奇怪，当时状态是正常的啊"
问题：状态在服务器内存里，重启就没了，查不到
```

**无状态的好处**

```text
用户说"我的请求失败了"
服务器查日志："请求A，Token是XXX，参数是YYY"
问题：所有信息都在请求里，日志记得清清楚楚
```

**比喻**：

| 有状态                                     | 无状态                                 |
| ------------------------------------------ | -------------------------------------- |
| 快递丢了，查快递员的记忆："我记得放门口了" | 快递丢了，查快递单号：每个节点都有记录 |
| 记忆靠不住                                 | 单据靠得住                             |

---

### 无状态的“代价”是什么？

| 代价                    | 说明                         | 怎么解决                         |
| ----------------------- | ---------------------------- | -------------------------------- |
| 每次请求都要带 Token    | 请求稍微大一点               | Token 一般很小，影响不大         |
| 服务器不能"主动推送"    | 服务器不能主动发消息给客户端 | 用 WebSocket 等长连接技术        |
| 敏感信息不能放 Token 里 | Token 可能被截获             | Token 只存身份，敏感信息服务器查 |

**总结：代价很小，好处很多。**

---

### 一句话记住

**无状态 = 服务器不记人，每次请求自带"身份证"。**

**为什么要这样？**

- **服务器可以随便复制、随便挂、随便扩**
- **请求可以随便重试、随便缓存、随便调试**
- **系统更可靠、更 scalable、更容易维护**

**有状态 = 服务器背着一个"记忆本"，越背越重，最后压垮。**  
**无状态 = 服务器轻装上阵，来一个处理一个，处理完就忘。**
