## 总结：

**GET 是”只读操作“，就像”看橱窗“，谁都可以看、能重复看、能自动看。如果用 GET 修改数据，等于把”转账按钮“放在橱窗里路人都能 accidentally 把你的钱转走！**

**POST 是”写入操作“，像”签字确认“，必须主动操作才会执行。**

**核心原因：GET 会被”自动执行“**

## 技术上的”连锁反应“

1.**缓存系统会”帮倒忙“**

| 场景                          | 后果                             |
| ----------------------------- | -------------------------------- |
| 你用 GET 删除数据 `?delete=1` | 运营商/浏览器会**缓存这个请求**  |
| 下次你再访问                  | 直接从缓存返回，**不经过服务器** |
| 或者缓存被其他人命中          | **别人的数据也被删了！**         |

**POST 不会被缓存**，每次都会经过服务器确认。

2.**浏览器”预加载“会闯祸**

| 场景                                             | 后果                    |
| ------------------------------------------------ | ----------------------- |
| 浏览器为了加速，会**提前加载**它觉得你要点的链接 | 看到 GET 链接就提前执行 |
| 你还没点"删除"按钮                               | 浏览器已经帮你删了      |
| 你想撤销                                         | 来不及了                |

**POST**不会被预加载，必须你主动点击。

3.**日志系统会”泄露秘密“**

| GET                                    | POST                                   |
| -------------------------------------- | -------------------------------------- |
| 参数在网址里，**所有经过的地方都记录** | 参数在请求身体里，**只有服务器能看到** |
| 浏览器历史记录里有 `?password=123456`  | 历史记录里只有干净的网址               |
| 公司网络日志、运营商日志全能看到       | 相对安全                               |
